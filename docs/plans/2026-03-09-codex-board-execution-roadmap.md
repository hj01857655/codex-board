# codex-board 执行路线图（2026-03-09）

## 1. 方向锁定（不偏航）

本阶段只做三类事情：

1. 批量操作可观测性（失败后能快速定位问题）
2. 审计与导出一致性（UI 看到什么，导出就有什么）
3. 回归防护（关键行为有测试，避免修复后再次退化）

明确不做：

1. 新增复杂功能入口
2. 增加与核心运维无关的 UI 装饰
3. 改动后端接口契约

---

## 2. 当前基线（已完成）

已落地并推送：

1. 凭证重命名时保留测试结果与选中状态（含单测）
2. 最近任务审计面板改为紧凑可展开（减少占屏）
3. 批量任务摘要支持失败项示例提示
4. BulkActionBar 与 CredentialTabs 失败提示文案逻辑统一

统一验证基线：

1. `pnpm test`
2. `pnpm build`

---

## 3. 后续待办（按优先级）

### P0：审计导出一致性补齐

目标：导出 JSON 中保留与页面一致的失败提示细节，避免“页面可见、导出丢失”。

实施项：

1. 审核 task audit record 的 `detail` 生成点（scan / bulk / smart）
2. 确认导出逻辑不截断关键 detail
3. 增加 1 条针对导出内容结构的测试（最小覆盖）

验收标准：

1. 页面显示失败示例时，导出 JSON 同样可见
2. `pnpm test` 通过
3. `pnpm build` 通过

### P1：失败原因分桶（不加新入口）

目标：失败不仅有“数量”，还要有“类型占比”（网络/鉴权/服务端/未知）。

实施项：

1. 在现有批量执行链路中增加失败类型归类（只在摘要/detail中展示）
2. 复用现有 summary 渲染区域，不新增按钮/面板
3. 补 2-3 条纯函数测试覆盖分桶规则

验收标准：

1. 一次批量失败后可看到类型分布
2. 旧流程交互不变
3. `pnpm test` + `pnpm build` 通过

### P2：批量失败快速重试（慎做）

目标：对失败集合做一次性重试，减少人工逐条补操作。

前置条件：

1. P0 与 P1 已完成且稳定
2. 失败集合有可靠来源（内存+审计一致）

实施项（最小化）：

1. 仅在现有批量区域提供“重试失败项”动作
2. 默认沿用当前并发策略和进度展示
3. 不引入新全局状态模型

验收标准：

1. 可针对最近一次失败集合重试
2. 失败/成功计数正确回写摘要与审计
3. `pnpm test` + `pnpm build` 通过

---

## 4. 每次提交必须满足

1. 只做一个明确目标（单主题提交）
2. 提交信息可读：`feat/fix/test/docs(scope): ...`
3. 附带可执行证据：测试与构建结果
4. 与本路线图边界冲突的需求，先确认再做

---

## 5. 暂停线（防止做偏）

出现以下任一情况立即暂停并回到路线图确认：

1. 需要新增多个入口或跨页面状态改造
2. 需要修改后端协议
3. 需求无法通过“现有 UI 区域最小改动”完成
